#1Silent cron failure — zero alerting
½ day owner: platform
Symptom
Modal crons run_pdf_extraction and run_browser_extraction failed silently from April 8 → April 18 (10 days). Nobody noticed because there is no external alert path.
Impact
Complete invisible outage. No new fees extracted, no errors surfaced. Could recur tomorrow on any other scheduled job (Knox, Darwin, ingest) and we wouldn't know.
Root cause
workers_last_run is a write-after-success dashboard marker. When a job crashes before writing it, the marker just stops updating — no error is raised anywhere visible.
Fix
  1. Add a "last-success freshness" check to the /admin/pipeline page (red banner if any job is stale >24h).
  2. Wire Modal's built-in webhook for function failures to a Slack channel or email alias.
  3. Add a cron-of-crons that queries workers_last_run and posts a health summary at 07:00 UTC daily.

#4RLS bypass — service_role is single blast radius
2 weeks owner: security
Symptom
Tier tables (fees_raw, fees_verified, fees_published) have RLS enabled but zero PostgREST policies. Only service_role can read/write. One key = total access.
Impact
If SUPABASE_SERVICE_ROLE_KEY leaks — via worktree commit, CI log, accidental console print, compromised developer laptop — the attacker has full read/write across every published fee and agent message. No defense in depth.
Root cause
RLS was enabled as a checkbox requirement for Supabase Auth compatibility. Proper per-role policies were deferred because "service_role handles everything."
Fix
  1. Design RLS policies per role: admin, analyst, premium, agent_runtime, public_api.
  2. Move agent writes off service_role onto a scoped agent_runtime role that can only INSERT into tier tables and agent_messages, not DELETE.
  3. Move public-API reads onto an anon role with per-table SELECT-only policies (already-published data only).
  4. Rotate service_role key after the new policies are in place.

#5Modal Starter 5-cron cap — squeeze-scheduling already in use
1–3 days owner: platform
Symptom
Modal Starter tier limits to 5 cron jobs. modal_app.py has explicit comments about squeezing post-processing onto an every-minute dispatcher with a marker table to gate work. Adding any new scheduled agent is blocked.
Impact
Can't add: weekly ingest-GLEIF, daily fed-data-refresh, hourly health-check cron, per-agent retry loops. Every new need has to piggyback on the dispatcher with its own marker-table gate — fragile.
Fix
  1. Option A: Upgrade Modal tier to remove the cap (~$50/mo).
  2. Option B: Build a proper meta-scheduler — one cron that reads a scheduled_jobs table and dispatches per window. Unlocks unlimited virtual crons.
  3. Migrate the existing dispatcher hack into the meta-scheduler pattern so it's a proper framework not a workaround.

#6No rollback path for fees_published
1 week owner: data
Symptom
If Darwin + Knox collude to publish a bad batch (e.g. a parser regression drops a zero), there is no DELETE path, no tombstone, no unpublish flow. lineage_ref lets us identify offenders but not remove them.
Impact
Bad data in the public index is permanent until manually nuked. Customer-facing inaccuracy risk. Reputation cost higher than the fix cost.
Fix
  1. Add retracted_at + retraction_reason columns to fees_published.
  2. Create a retract_fee_published(id, reason) function that soft-deletes (sets retracted_at) and triggers a revalidatePath('/fees/:id') cache bust.
  3. Build admin UI at /admin/retractions with per-fee retract button + audit trail.
  4. Filter retracted_at IS NULL in every public-facing query.

#7Knox-rejected fees have no human review UI
2–3 days owner: admin UX
Symptom
When Knox rejects a fees_verified row, the row sits there with a darwin-accept and a knox-reject in agent_messages. No admin page shows these. 34 such rejects from today's session are invisible.
Impact
Knox rejects are a signal, not a verdict — often the fee is correct and Knox's rule is wrong. Without human adjudication, we lose genuine data and don't learn which Knox rules to tune.
Fix
  1. New page /admin/knox-rejects with queue of rows where knox-reject exists and no human override.
  2. Show: fee details, darwin's reasoning, knox's rejection reasons, peer-median context.
  3. Actions: approve anyway (writes human_accept message, republishes), confirm reject (writes human_confirm, sends feedback to Knox rules file).
  4. Weekly report: top Knox rules by override rate — candidates for tuning.

#8agent_messages grows unbounded
2–3 days owner: platform
Symptom
648 rows today. At full scale (100K fees/day × 2 msgs/fee) = 200K rows/day = 73M/year. No partitioning, no TTL, no archive policy.
Impact
Query performance degrades linearly. promote_to_tier3 SQL gate reads this table on every publish. At 10M+ rows without partitioning, daily publish takes minutes not seconds.
Fix
  1. Partition agent_messages by month using PARTITION BY RANGE (created_at).
  2. Add a monthly cron that archives partitions older than 90 days to cold storage (R2 export).
  3. Create a summary table agent_message_stats that aggregates per-fee-verified-id so the SQL gate doesn't need the raw table.

#9Schema drift risk — no migration tracking table
½ day owner: platform
Symptom
migrations/ has files dated 2026-04-06 → 2026-05-10 (future-dated from today). No way from the repo to know which are applied vs staged. Developers running scripts/migrate-schema.sql get a different schema than those who run the dated migrations.
Impact
Local/CI/prod schema drift. A test that passes locally can fail in prod if a migration wasn't applied. Onboarding new devs is a minefield.
Fix
  1. Create schema_migrations(filename TEXT PK, applied_at TIMESTAMPTZ) table.
  2. Wrap scripts/apply-migrations.mjs to insert a row on success.
  3. Add a GET /api/admin/schema-state route that lists applied vs pending migrations and surface on /admin.
  4. Document in README: canonical bootstrap is migrate-schema.sql; incrementals go through dated migrations.

1–2 days owner: data
Symptom
One column holds FDIC CERT (banks) and NCUA charter # (credit unions), disambiguated only by source. Both are small integers; a CERT and a charter could collide on value.
Impact
Any JOIN that forgets source produces silent wrong-bank matches. Cross-source reports could show a bank's fees under a CU's name.

Closed by the identity-map migration (§6 of architecture doc — 20260418_institution_identity.sql). Backfill ncua_charter_id, then migrate 20+ queries using source='ncua' AND cert_number=... to use the new column, then drop the overloading.

#11No URL re-validation for stale fee_schedule_url
2 days owner: crawler
Symptom
URLs are set once (by Magellan or institution seed) and we keep hammering them. If a bank moves their fee schedule to a new URL, we extract from a 301/404 page for months.
Impact
Silently stale fees. An institution could change prices and we'd publish last year's data. Known to affect ~9.5% of URLs per the URL health check memory.
Fix
  1. Quarterly "URL health" cron that HEAD-requests every fee_schedule_url and flags 3xx/4xx/5xx.
  2. On flag, re-run Magellan URL discovery for that institution before the next crawl.
  3. Surface stale-URL count on /admin/coverage.

#13Darwin re-classification is lossy — no history
2 days owner: data
Symptom
If Darwin changes its mind on a fees_verified row (e.g. after a taxonomy update), canonical_fee_key gets overwritten in place. No history of prior classifications.
Impact
Cannot audit why a fee moved categories. Cannot reproduce a historical report. Trust erodes when "the same fee" is labeled differently across snapshots.
Fix
  1. Add classification_history table: (fee_verified_id, old_key, new_key, darwin_run_id, changed_at).
  2. Trigger on UPDATE of canonical_fee_key writes a history row.
  3. Expose on /admin/fees/[id]/history.

#14No backoff on repeatedly-failing crawls
½ day owner: crawler
Symptom
consecutive_failures is tracked on crawl_targets but no exponential-backoff logic reads it. A URL returning 429/5xx gets hit at the same cadence every day indefinitely.
Impact
Wasted crawl budget on dead URLs. Anti-bot systems escalate (CAPTCHA, IP ban) making the URL harder to fix later. Potential TOS violations on some hosts.
Fix
  1. Skip-next-crawl logic: consecutive_failures >= 3 → skip for 3 days, >= 7 → skip for 14 days, >= 14 → mark status='dormant' and alert.
  2. Surface dormant list on /admin/coverage so humans can investigate.
