  <action>
Create `.github/workflows/e2e.yml` with the following structure:

**Triggers:**
- `schedule`: two cron entries
  - Nightly fast: `cron: '0 0 * * *'` (00:00 UTC daily)
  - Weekly full: `cron: '0 2 * * 0'` (02:00 UTC Sunday)
- `workflow_dispatch`: with a single input `mode` (type: choice, options: [fast, full], default: fast)

**Concurrency:** Use `concurrency: { group: "e2e-${{ github.ref }}", cancel-in-progress: true }` to prevent stacked runs.

**Job: `e2e-fast`**
- `if`: runs when trigger is the nightly schedule OR (workflow_dispatch AND inputs.mode == 'fast')
- Detect via: `github.event_name == 'schedule' && github.event.schedule == '0 0 * * *'` for schedule leg; combine with dispatch leg using `||`
- `runs-on: ubuntu-latest`
- `timeout-minutes: 30`
- Steps:
  1. `actions/checkout@v4`
  2. `actions/setup-python@v5` with `python-version: '3.12'`
  3. `pip install -r requirements.txt pytest pytest-timeout`
  4. `pytest -m "e2e and not llm and not slow" --timeout=600 -v` (per D-01)
- No secrets needed

**Job: `e2e-full`**
- `if`: runs when trigger is the weekly schedule OR (workflow_dispatch AND inputs.mode == 'full')
- Detect via: `github.event_name == 'schedule' && github.event.schedule == '0 2 * * 0'` for schedule leg
- `runs-on: ubuntu-latest`
- `timeout-minutes: 90`
- `env`: `ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}` (per D-03)
- Steps:
  1. `actions/checkout@v4`
  2. `actions/setup-python@v5` with `python-version: '3.12'`
  3. `pip install -r requirements.txt pytest pytest-timeout`
  4. `pytest -m e2e --timeout=600 -v` (per D-01: full mode runs all e2e including llm and slow)

**Implementation notes:**
- Use `name: e2e` as the workflow name
- Do NOT use `on:` with a plain `schedule:` top-level if dispatch is also needed — combine both under `on:` correctly (YAML map, not list)
- The `if:` condition for schedule-triggered jobs must check `github.event.schedule` to distinguish nightly from weekly — GitHub passes the matched cron string in this field
- Keep both jobs independent (no `needs:`) so they can run in parallel when both are dispatched
- No matrix strategy needed — single-platform run is sufficient
- Do not add `continue-on-error: true`; failures must be visible
