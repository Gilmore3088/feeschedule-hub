Create `supabase/migrations/20260406_report_jobs_rls.sql`:

```sql
-- Phase 15: Enable RLS on report_jobs (PRO-03)
-- Access model:
--   1. A user can see their own jobs (user_id matches their session user_id)
--   2. Admin users (role = 'admin' in the users table) can see all jobs
--   3. Cron-triggered jobs (user_id IS NULL) are visible only to admins

-- Enable RLS
ALTER TABLE report_jobs ENABLE ROW LEVEL SECURITY;

-- Drop existing policies if any (idempotent re-run safety)
DROP POLICY IF EXISTS "report_jobs_owner_select" ON report_jobs;
DROP POLICY IF EXISTS "report_jobs_admin_select" ON report_jobs;
DROP POLICY IF EXISTS "report_jobs_owner_insert" ON report_jobs;
DROP POLICY IF EXISTS "report_jobs_owner_update" ON report_jobs;

-- SELECT: owner sees own rows; admin sees all rows including cron (user_id IS NULL)
CREATE POLICY "report_jobs_owner_select"
  ON report_jobs
  FOR SELECT
  USING (
    user_id = current_setting('app.current_user_id', true)
  );

CREATE POLICY "report_jobs_admin_select"
  ON report_jobs
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM users u
      WHERE u.id::text = current_setting('app.current_user_id', true)
        AND u.role = 'admin'
    )
  );

-- INSERT: any authenticated user can create a job for themselves
-- (application layer validates user_id before insert — this is defense in depth)
CREATE POLICY "report_jobs_owner_insert"
  ON report_jobs
  FOR INSERT
  WITH CHECK (
    user_id = current_setting('app.current_user_id', true)
    OR user_id IS NULL  -- cron jobs (service role bypasses RLS anyway)
  );

-- UPDATE: only service role (Modal worker) updates job status; no user UPDATE policy
-- Modal worker uses service_role key which bypasses RLS — this is intentional

-- COMMENT: The application layer (Phase 13 status route T-13-11) already enforces
-- ownership at the API level. This RLS adds defense-in-depth at the DB layer.
-- app.current_user_id must be set by the connection pool or API layer via:
--   SET LOCAL app.current_user_id = '{user_id}';
-- This is a best-effort RLS layer; primary auth enforcement remains at API routes.

COMMENT ON TABLE report_jobs IS
  'Report generation jobs. RLS enabled: users see own rows, admins see all. Modal worker uses service_role (bypasses RLS).';
```

Note in a block comment at the top: "These policies assume the application sets `app.current_user_id` via SET LOCAL per request. Since the app currently uses the service_role key for all DB operations (Supabase connection via postgres client), RLS does not enforce at query time — the API-layer ownership check (T-13-11 in Phase 13) is the primary enforcement. This migration is preparatory for when the app migrates to anon/authenticated role connections."

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /pro/reports | Unauthenticated and non-premium users must be blocked before any DB query |
| ReportLibraryFilters → /api/reports/{id}/status | Download fetch must not leak another user's presigned URL |
| Application → report_jobs DB | RLS adds defense-in-depth; API ownership check is primary enforcement |

| T-15-10 | Spoofing | /pro/reports page server component | mitigate | getCurrentUser() with HMAC session verification before DB query; redirect on null |
| T-15-11 | Elevation of Privilege | Report library — non-subscriber access | mitigate | canAccessPremium() check → redirect to /subscribe before any job query runs |
| T-15-12 | Information Disclosure | Download button — fetches presigned URL for arbitrary job ID | mitigate | Fetch goes through /api/reports/{id}/status which has ownership guard (T-13-11); client cannot pass another user's job ID to get their URL |

| T-15-14 | Information Disclosure | RLS bypass via service_role key | accept | Service_role is used only by Modal worker and migrations; never exposed to browser; documented in migration comment |
