Create `src/lib/report-engine/assemble-and-render.ts`. This is a pure async function
that accepts (reportType, params, jobId) and returns a complete HTML string.

**Function signature:**
```typescript
export async function assembleAndRender(
  reportType: ReportType,
  params: Record<string, unknown>,
  jobId: string,
): Promise<string>
```

**Step-by-step implementation:**

1. Import getSql from `@/lib/crawler-db/connection` for job status updates.
2. Update job status to 'assembling' at entry:
   `await sql\`UPDATE report_jobs SET status = 'assembling' WHERE id = ${jobId}\``
3. Switch on reportType with four branches:

**Branch: national_index**
- Call `assembleNationalQuarterly()` → payload
- Run Hamilton calls in parallel using Promise.allSettled() (D-08: graceful degradation):
  - executive_summary: `generateSection({ type: 'executive_summary', title: 'National Fee Landscape — Key Findings', data: { total_institutions: payload.total_institutions, total_bank_institutions: payload.total_bank_institutions, total_cu_institutions: payload.total_cu_institutions, category_count: payload.categories.length, top_categories: payload.categories.slice(0, 5).map(c => ({ name: c.display_name, median: c.median_amount, count: c.institution_count })) } })`
  - national_index: `generateSection({ type: 'trend_analysis', title: 'Fee Distribution Across 49 Categories', data: { categories: payload.categories.map(c => ({ name: c.display_name, median: c.median_amount, p25: c.p25_amount, p75: c.p75_amount, count: c.institution_count, maturity: c.maturity_tier })) } })`
  - charter_analysis (conditional — only if some category has bank_count > 0 and cu_count > 0): `generateSection({ type: 'peer_comparison', title: 'Banks vs. Credit Unions — Where Charter Type Drives Fee Divergence', data: { categories: payload.categories.filter(c => c.bank_count > 0 && c.cu_count > 0).slice(0, 10).map(c => ({ name: c.display_name, bank_median: c.bank_median, cu_median: c.cu_median })) } })`
  - district_context (conditional — only if payload.district_headlines.length > 0): `generateSection({ type: 'regional_analysis', title: 'Fed District Economic Conditions Influencing Fee Pressure', data: { headlines: payload.district_headlines } })`
- Validate each settled result with `validateNumerics()`. If validation fails (invented numbers), log a warning with `console.warn('[assembleAndRender] numeric validation failed:', section, result.inventedNumbers)` but still include the narrative (D-07 says validate and log, not reject).
- If a Hamilton call rejects (PromiseSettledResult status === 'rejected'), use a null narrative for that section — the template handles undefined/absent narrative keys gracefully (the conditional checks in templates already guard this).
- Render: `renderNationalQuarterlyReport({ data: payload, narratives: { executive_summary, national_index, charter_analysis?, district_context? } })`

**Branch: state_index**
- Extract stateCode: `typeof params.state_code === 'string' ? params.state_code.toUpperCase() : 'US'`
- Build stateName from an inline STATE_NAMES map (all 50 states + DC). Fallback: stateCode.
- The state template is a stub — it does NOT take fee data. Just call:
  `renderStateFeeIndexReport({ stateCode, stateName, generatedAt: new Date().toISOString().slice(0, 10) })`
- No Hamilton calls for state_index (the template doesn't accept narratives — it's a placeholder).

**Branch: monthly_pulse**
- Call `assembleMonthlyPulse()` → payload
- Run 1 Hamilton call:
  - pulse_overview: `generateSection({ type: 'overview', title: \`Fee Market Movement — ${payload.period_label}\`, data: { period_label: payload.period_label, total_movers: payload.total_movers, total_categories_tracked: payload.total_categories_tracked, movers_up: payload.movers_up.slice(0, 5), movers_down: payload.movers_down.slice(0, 5) }, context: 'Write a 1-2 paragraph executive summary. 250 words maximum.' })`
- Validate with validateNumerics(). Warn on failure, include narrative regardless.
- If Hamilton rejects, fallback: `{ narrative: 'Market movement data is presented in the tables below.', wordCount: 10, model: 'fallback', usage: { inputTokens: 0, outputTokens: 0 } }`
- Render: `renderMonthlyPulseReport({ data: payload, narratives: { pulse_overview } })`

**Branch: peer_brief**
- Extract filters from params:
  ```typescript
  const filters: PeerCompetitiveFilters = {
    charter_type: typeof params.charter_type === 'string' ? params.charter_type : undefined,
    asset_tiers: Array.isArray(params.asset_tiers) ? params.asset_tiers as string[] : undefined,
    fed_districts: Array.isArray(params.fed_districts) ? params.fed_districts as number[] : undefined,
  };
  ```
- Call `assemblePeerCompetitivePayload(filters)` → payload
- Run 2 Hamilton calls in parallel (Promise.allSettled):
  - executive_summary: use the first section from payload.sections where include=true and section_type='executive_summary': `generateSection({ type: 'executive_summary', title: section.title, data: section.data })`
  - featured_fees: use the section where section_type='peer_competitive': `generateSection({ type: 'peer_competitive', title: section.title, data: section.data })`
- Validate each. Warn on failure.
- Fallback for each if rejected: `{ narrative: 'Competitive analysis data is presented in the tables below.', wordCount: 8, model: 'fallback', usage: { inputTokens: 0, outputTokens: 0 } }`
- Render: `renderPeerCompetitiveReport({ data: payload.data, narratives: { executive_summary, featured_fees } })`

**Default branch:** throw new Error(`Unknown report_type: ${reportType}`)

**Error handling:** Wrap the entire function body in try/catch. On any unhandled error, update
job status to 'failed' before re-throwing:
```typescript
await sql`UPDATE report_jobs SET status = 'failed', error = ${message} WHERE id = ${jobId}`;
throw err;
```

**STATE_NAMES inline map** (include all 50 states + DC as a `Record<string, string>` constant at the top of the file, e.g. `{ WY: 'Wyoming', TX: 'Texas', CA: 'California', ... }`).

Do NOT call `db.close()` — getSql() returns the singleton postgres client (never close).

  <name>Task 2: Update generate route to call assembleAndRender() before Modal trigger</name>
  <files>src/app/api/reports/generate/route.ts</files>
  <action>
Update the fire-and-forget Modal trigger block in the generate route (lines 129–145).

The current flow is: insert job → fire Modal with `html: ''` → return 202.

The new flow is: insert job → return 202 immediately → background: assembleAndRender() →
update status to 'rendering' → fire Modal with real HTML.

**Why return 202 first, then run assembly in background:** assembleAndRender() makes DB
queries and Claude API calls (30-90 seconds). The client must not wait. The existing fire-and-
forget `.catch()` pattern is the right structure — extend it.

**Exact changes:**

1. Add import at top of file (after existing imports):
   ```typescript
   import { assembleAndRender } from '@/lib/report-engine/assemble-and-render';
   ```

2. Replace the existing fire-and-forget block (the `if (!modalUrl)` block through the
   closing `}`  on line ~145) with:

```typescript
// Fire-and-forget: assemble HTML then trigger Modal — do NOT await (job tracked via polling)
if (!modalUrl) {
  console.warn('[reports/generate] MODAL_REPORT_URL not configured — skipping Modal trigger');
} else {
  // Non-blocking: intentionally not awaited
  (async () => {
    try {
      // D-09: assembleAndRender updates status to 'assembling' internally
      const html = await assembleAndRender(validatedType, validatedParams, jobId);

      // Update status to 'rendering' before Modal call
      const sql = getSql();
      await sql`UPDATE report_jobs SET status = 'rendering' WHERE id = ${jobId}`;

      await fetch(modalUrl, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          job_id: jobId,
          html,
          report_type: validatedType,
        }),
      });
    } catch (err: unknown) {
      console.error(
        '[reports/generate] assembly or Modal trigger failed:',
        err instanceof Error ? err.message : String(err),
      );
      // assembleAndRender() already updates status to 'failed' on its own errors.
      // This catch handles unexpected failures outside that scope.
    }
  })();
}
```

3. The `return NextResponse.json({ jobId }, { status: 202 })` line must remain BEFORE the
   background IIFE — do not move it. The IIFE runs after the return is already dispatched.

Note: The `sql` variable declared earlier (line 108) is used for the INSERT. The new `sql`
inside the async IIFE is a fresh reference from getSql() — this is correct (postgres client
is a singleton; multiple references are safe).
