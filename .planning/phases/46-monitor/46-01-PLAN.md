## Objective

Replace the `/pro/monitor` stub with a fully functional continuous surveillance screen. The Monitor screen is the default Pro landing page — users see it first every session. It shows the current system state (stable/watch/worsening), a priority alert card for the top active alert, a reverse-chronological signal feed timeline, a watchlist panel with institution status indicators, and a floating chat overlay that lets users ask Hamilton questions without navigating away.

Signal data is pre-seeded for v8.0 demo. The automated signal pipeline is deferred.

---

## Success Criteria

1. The status strip displays the current system state (stable/watch/worsening) and a signal count — values reflect seeded signal data, not hardcoded placeholders
2. The Priority Alert card shows the top alert with severity badge, a one-sentence impact description, and a "Recommended next move" action link
3. The Signal Feed timeline shows signals in reverse-chronological order — each signal has a timestamp, institution or category label, and deviation description
4. The Watchlist panel shows at least one tracked institution with a renewal or review status indicator — the institution can be added and removed from the watchlist
5. The floating chat overlay opens without navigating away from Monitor and accepts a question — Hamilton's response appears in the overlay without disrupting the underlying signal feed
6. When no signals exist, the Monitor shows a guided onboarding empty state (not a blank screen)

---

## Threat Model

<threat_model>
**Screen:** `/pro/monitor` — authenticated Pro subscriber only

**Threats:**
- **T1 (Medium):** Unauthorized access to signals/alerts — user A reads user B's priority alerts. Mitigation: `hamilton_priority_alerts` queries always include `WHERE user_id = ${user.id}` filter. Layout enforces premium gate via `canAccessPremium()`.
- **T2 (Low):** Floating chat prompt injection — user crafts monitor-mode message that elicits recommendation language. Mitigation: same system prompt guard applied in analyze mode (no "recommended position" language) applies to monitor mode.
- **T3 (Low):** XSS via Hamilton-generated chat content. Mitigation: use React controlled rendering — never `dangerouslySetInnerHTML` without sanitization.
- **T4 (Low):** Watchlist mutation (add/remove) lacks auth check — any user can modify another user's watchlist. Mitigation: Server Actions for watchlist mutations include `WHERE user_id = ${user.id}` in all UPDATE/DELETE operations.
- **T5 (Low):** Cost runaway from floating chat abuse. Mitigation: daily cost circuit breaker already in `/api/research/hamilton/route.ts` ($50/day limit).

**Residual risk:** Low. Auth gate, cost circuit breaker, and per-user DB scoping are all pre-existing.
</threat_model>

### T-46-02 — Add monitor data fetcher

**File:** `src/lib/hamilton/monitor-data.ts` (new)

Create typed data fetcher for the Monitor screen. Export:

```typescript
export interface MonitorPageData {
  status: {
    overall: "stable" | "watch" | "worsening";
    newSignals: number;
    highPriorityAlerts: number;
  };
  topAlert: AlertEntry | null;
  signalFeed: SignalEntry[];
  watchlist: WatchlistEntry[];
}

export interface WatchlistEntry {
  id: string;
  institutionId: string;
  displayName: string;
  status: "current" | "review_due" | "unknown";
}

export async function fetchMonitorPageData(userId: number): Promise<MonitorPageData>
```

Implementation notes:
- `signalFeed`: fetch last 20 signals from `hamilton_signals` ordered `DESC` by `created_at` — reuse `fetchRecentSignals` pattern from `home-data.ts`
- `topAlert`: fetch the top 1 active alert for user from `hamilton_priority_alerts` JOIN `hamilton_signals` ordered by severity then `created_at DESC`
- `status.overall`: derived from signal severity distribution — if any high-severity signal in last 24h → "worsening"; if any medium → "watch"; else "stable"
- `status.newSignals`: count of signals created in last 24h
- `status.highPriorityAlerts`: count of active alerts with severity = 'high' for this user
- `watchlist`: fetch from `hamilton_watchlists` for user — return institution_ids as `WatchlistEntry[]` with `status: "unknown"` for v8.0 (status derivation is deferred)
- All DB calls wrapped in try/catch returning empty/null on failure

**Acceptance:** `fetchMonitorPageData(userId)` returns typed data with no runtime errors when tables are empty or seeded.

### T-46-09 — Wire monitor page

**File:** `src/app/pro/(hamilton)/monitor/page.tsx`

Replace the stub. This is a Server Component that:
1. Gets current user via `getCurrentUser()`
2. Calls `seedMonitorData(user.id)` to ensure demo data exists
3. Calls `fetchMonitorPageData(user.id)` for all screen data
4. Renders the full Monitor layout

```tsx
import type { Metadata } from "next";
import { getCurrentUser } from "@/lib/auth";
import { seedMonitorData } from "@/lib/hamilton/seed-monitor-data";
import { fetchMonitorPageData } from "@/lib/hamilton/monitor-data";
import { StatusStrip } from "@/components/hamilton/monitor/StatusStrip";
import { PriorityAlertCard } from "@/components/hamilton/monitor/PriorityAlertCard";
import { SignalFeed } from "@/components/hamilton/monitor/SignalFeed";
import { WatchlistPanel } from "@/components/hamilton/monitor/WatchlistPanel";
import { FloatingChatOverlay } from "@/components/hamilton/monitor/FloatingChatOverlay";
import { redirect } from "next/navigation";

export const metadata: Metadata = { title: "Institutional Monitor" };

export default async function MonitorPage() {
  const user = await getCurrentUser();
  if (!user) redirect("/login");

  await seedMonitorData(user.id);
  const data = await fetchMonitorPageData(user.id);

  return (
    <>
      {/* Status strip — full width above content */}
      <StatusStrip status={data.status} />

      {/* Main content grid */}
      <div style={{ display: "grid", gridTemplateColumns: "1fr 320px", gap: "1.5rem", padding: "1.5rem", maxWidth: "1280px", margin: "0 auto" }}>
        {/* Left column: Priority Alert + Signal Feed */}
        <div style={{ display: "flex", flexDirection: "column", gap: "1.25rem" }}>
          <PriorityAlertCard alert={data.topAlert} />
          <SignalFeed signals={data.signalFeed} />
        </div>

        {/* Right rail: Watchlist */}
        <aside>
          <WatchlistPanel entries={data.watchlist} userId={user.id} />
        </aside>
      </div>

      {/* Floating chat overlay — fixed position, renders outside grid */}
      <FloatingChatOverlay userId={user.id} />
    </>
  );
}
```

Page does NOT use ISR revalidation — fresh data on every load (per CONTEXT D-11 pattern from home screen).

**Acceptance:** Navigating to `/pro/monitor` renders all four panels (StatusStrip, PriorityAlertCard, SignalFeed, WatchlistPanel) with seeded data. No TypeScript compile errors.
