const ProfileSchema = z.object({

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client form -> Server action | User-submitted profile data crosses from untrusted client to server |
| Avatar dropdown -> Settings route | Navigation only, no trust boundary crossed |
| Server action -> Database | Parameterized SQL prevents injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-01 | Spoofing | updateInstitutionProfile | mitigate | Server action calls getCurrentUser() to verify authenticated session before any DB write |
| T-41-02 | Tampering | Profile form data | mitigate | Zod schema validates all fields (enum constraints, string length limits, integer range 1-12 for district) |
| T-41-03 | Information Disclosure | User data in avatar dropdown | accept | Only shows authenticated user's own display_name and email — no cross-user data |
| T-41-04 | Elevation of Privilege | Profile update allows role change | mitigate | Server action only updates institution profile columns — role, password, email are NOT in the update SET clause |
| T-41-05 | Injection | SQL injection via profile fields | mitigate | All queries use parameterized sql template literals from postgres library |
| T-41-06 | Tampering | IDOR — updating another user's profile | mitigate | WHERE id = currentUser.id — user can only modify their own row |
</threat_model>
