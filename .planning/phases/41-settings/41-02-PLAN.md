<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client form -> createPeerSet action | User-submitted peer set configuration crosses to server |
| Client -> removePeerSet action | Deletion request with peer set ID |
| Server -> Stripe API | Billing portal session creation (if Stripe configured) |
| Server -> hamilton_* tables | Read-only COUNT queries for snapshot |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-07 | Spoofing | createPeerSet | mitigate | getCurrentUser() check before any peer set write |
| T-41-08 | Tampering | removePeerSet IDOR | mitigate | deletePeerSet takes userId parameter — only deletes rows where created_by matches |
| T-41-09 | Information Disclosure | Intelligence snapshot | mitigate | COUNT queries scoped to WHERE user_id = currentUser.id |
| T-41-10 | Elevation of Privilege | Stripe portal access | mitigate | Portal session created with user's own stripe_customer_id only |
| T-41-11 | Denial of Service | Peer set spam | accept | No rate limit on peer set creation — low risk for authenticated Pro users; natural limit is UX friction |
| T-41-12 | Tampering | Feature toggle state | accept | Toggles are client-side visual only in v8.0 — no server persistence, no security impact |
</threat_model>
