# Phase 60.1 Plan 02: SQLite Retirement for Pipeline Tables — Summary

**One-liner:** Formal retirement of SQLite for pipeline tables with `require_postgres()` fail-fast guard planted at all six pipeline-touching entrypoints and a Postgres structural probe added to `modal_preflight.py`.

## What Was Built

### Task 1 — `fee_crawler/db.py`

Added three artifacts:

1. `_PIPELINE_TABLES_REQUIRING_POSTGRES` — module-level tuple listing the six pipeline-only tables (`jobs`, `platform_registry`, `cms_confidence`, `document_r2_key`, `document_type_detected`, `doc_classification_confidence`).

2. `require_postgres(reason: str)` — guard helper that raises `RuntimeError("SQLite not supported for pipeline tables; set DATABASE_URL ...")` when `_DATABASE_URL` is falsy. The message includes the caller-supplied reason and a pointer to the phase directory.

3. Updated `Database` class docstring to state the SQLite/Postgres table boundary explicitly.

4. Added `# LEGACY: SQLite path covers ...` comment at the top of `_init_tables()` so the intent is clear at the call site.

The SQLite CREATE TABLE statements are untouched. Legacy tables (`crawl_targets`, `crawl_runs`, `crawl_results`, `extracted_fees`) continue to work under SQLite.

### Task 2 — `fee_crawler/modal_preflight.py`

The existing `preflight_e2e` function deliberately pops `DATABASE_URL` and runs against an isolated SQLite DB in `/tmp`. This isolation is intentional and was preserved intact.

Added `preflight_postgres()` as a new `@app.function` / `@modal.fastapi_endpoint(method="GET")` that:
- Calls `require_postgres("preflight_postgres validates pipeline table existence")` at its top
- Connects to `DATABASE_URL` via psycopg2 (already a runtime dependency)
- Runs a single `SELECT to_regclass(...)` query probing all six pipeline tables
- Raises `RuntimeError` listing any NULL (missing) tables
- Returns `{"status": "pass", "message": ..., "tables": [...]}` on success

Updated the module docstring to document both preflight modes and their deployment commands.

### Task 3 — Four pipeline-touching files

All four files in the audit Finding 2 evidence list received guards:

| File | Guard call | Pipeline tables accessed |
|------|-----------|--------------------------|
| `fee_crawler/workers/discovery_worker.py` | `require_postgres("discovery_worker requires pipeline tables (jobs / platform_registry)")` at top of `run()` | `jobs`, `platform_registry` |
| `fee_crawler/commands/crawl.py` | `require_postgres("crawl command requires pipeline tables (document_r2_key / document_type_detected / doc_classification_confidence / platform_registry)")` at top of `run()` | `document_r2_key`, `document_type_detected`, `doc_classification_confidence`, `platform_registry` |
| `fee_crawler/workers/data_integrity.py` | `require_postgres("data_integrity requires pipeline tables (jobs)")` at top of `run_checks()` | `jobs` |
| `fee_crawler/workers/daily_report.py` | `require_postgres("daily_report requires pipeline tables (jobs)")` at top of `generate_report()` | `jobs` |

Each file imports `require_postgres` from `fee_crawler.db`. No runtime logic was altered beyond the import and single call per entrypoint.

## Deviations from Plan

### Preserved Finding (pre-existing, out-of-scope)

**`test_call_report_scaling.py` collection error** — This test imports `_apply_ffiec_scaling` from `fee_crawler.commands.ingest_call_reports`, which does not exist (audit Finding 5). This is a pre-existing regression that is the subject of plan 60.1-05, not this plan. The error was present before any changes in this plan and is documented in `.planning/phases/60.1-audit-remediation-operational-reliability/60.1-CONTEXT.md` under "Phase 59 test regressions." It is out-of-scope for plan 60.1-02.

### Design Choice — Separate preflight_postgres endpoint

The plan said to call `require_postgres` "at the top of whatever entrypoint function the file exposes." The existing `preflight_e2e` entrypoint intentionally pops `DATABASE_URL` as part of its isolation contract (the comment says "ISOLATION: this function deliberately ignores DATABASE_URL"). Inserting `require_postgres` into `preflight_e2e` would break that isolation. Instead, a new `preflight_postgres()` endpoint was added. This satisfies the plan's intent (preflight validates Postgres assumptions) without breaking the SQLite isolation test.
