# Phase 60.1: Code Review Report — Operational Reliability Pass

**Reviewed:** 2026-04-15
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

Phase 60.1 substantially improves the operational reliability posture of the pipeline:
`run_checked` + `SubprocessFailed` correctly fail loudly on non-zero exits; `require_postgres`
is called before DB work in every changed worker entrypoint except one; `markJobFailed` is
properly wrapped and cannot crash the response handler; and the Stealth fallback raises a clear
`RuntimeError` rather than silently degrading. No hardcoded secrets or new SQL-injection surfaces
were found.

Three medium-severity issues require attention before this phase is considered clean:

1. `ingest_call_reports.py::run` has no `require_postgres()` guard — it will crash with a bare
   `KeyError` on `os.environ["DATABASE_URL"]` instead of a structured error when `DATABASE_URL`
   is absent, and the crash happens *after* a cursor is allocated.
2. The FFIEC balance-sheet columns (`ASSET`, `DEP`, `LNLSNET`) are stored raw (thousands) while
   `service_charge_income` and `other_noninterest_income` are now scaled to whole dollars — an
   inconsistent unit within the same row.
3. The migration 023 SQL guard (`< 100000000`) does not protect `other_noninterest_income` from
   double-application if the migration is re-run after the ingestor has already stored scaled
   values.

## Warnings

### WR-01: `ingest_call_reports.py::run` missing `require_postgres()` guard

**File:** `fee_crawler/commands/ingest_call_reports.py:386`
**Severity:** P1 major
**Issue:** Every other worker that touches pipeline tables calls `require_postgres()` before
opening a DB connection. `ingest_call_reports.run()` does not. Running without `DATABASE_URL`
raises an unstructured `KeyError: 'DATABASE_URL'` from deep inside psycopg2 (after the cursor
factory is constructed) instead of the informative `RuntimeError` that `require_postgres`
produces. This makes on-call triage harder and means the pattern is inconsistent.

**Fix:**
```python
def run(*, backfill: bool = False, from_year: int = 2010) -> None:
    from fee_crawler.db import require_postgres
    require_postgres("ingest_call_reports requires Postgres (institution_financials table)")
    conn = psycopg2.connect(os.environ["DATABASE_URL"])
    ...
```

### WR-02: Balance-sheet columns stored in thousands while income columns stored in whole dollars

**File:** `fee_crawler/commands/ingest_call_reports.py:197, 257`
**Severity:** P1 major
**Issue:** `_apply_ffiec_scaling` correctly multiplies `SC` (service charges) and `NONII`
(non-interest income) by 1000. `INTINC` and `EINTEXP` are also scaled at lines 323-324.
However `ASSET`, `DEP`, and `LNLSNET` are passed through `_safe_int` without scaling and
inserted into `total_assets`, `total_deposits`, and `total_loans`. The SQL comment in
migration 023 acknowledges balance-sheet columns are not currently ingested but they **are**
— they are in both upsert calls. This means a bank with $1B in assets will be recorded as
$1,000 (in thousands), while its service-charge income of $5M will be recorded as
$5,000,000,000 after scaling. Any ratio or comparison across these columns will be wrong.

**Fix (option A — scale all FDIC fields consistently):**
```python
# In _ingest_quarter, after safe_int extraction:
sc, nonii = _apply_ffiec_scaling("ffiec", sc_raw, nonii_raw)
asset = _safe_int(data.get("ASSET"))
dep   = _safe_int(data.get("DEP"))
loans = _safe_int(data.get("LNLSNET"))
# FDIC BankFind also reports balance sheet in thousands
asset = asset * 1000 if asset is not None else None
dep   = dep   * 1000 if dep   is not None else None
loans = loans * 1000 if loans is not None else None
```

**Fix (option B — extend `_apply_ffiec_scaling` to cover all five fields):**
Expand the helper signature to include asset/deposit/loan fields and apply the same
`* 1000 if source == "ffiec"` pattern.

Either option must also update migration 023 to back-fill the balance-sheet columns,
or at minimum update the comment so it stops saying "NOT currently ingested."

### WR-03: Migration 023 idempotency guard only covers `service_charge_income`

**File:** `scripts/migrations/023-fix-ffiec-scaling.sql:14-24`
**Severity:** P1 major
**Issue:** The `WHERE service_charge_income < 100000000` guard prevents double-application of
the `* 1000` factor to `service_charge_income` if the migration is re-run. But the guard does
**not** protect `other_noninterest_income` — the `CASE WHEN ... THEN oni * 1000` branch runs
regardless of the `WHERE` clause once `service_charge_income` passes. If, after the ingestor
runs (storing pre-scaled values of e.g. $500K = 500), the migration is applied again with the
same guard, the `service_charge_income` row would be skipped (already > 100000000 threshold? no
— only if it was already scaled before the migration), but `other_noninterest_income` is
updated unconditionally within that same `UPDATE`. The guard is asymmetric.

**Fix:**
```sql
WHERE source IN ('ffiec', 'fdic')
  AND service_charge_income IS NOT NULL
  AND service_charge_income < 100000000
  AND (other_noninterest_income IS NULL OR other_noninterest_income < 100000000);
```

### IN-01: `SubprocessFailed` used as a sentinel for non-subprocess failures in `ingest_data`

**File:** `fee_crawler/modal_app.py:226-231`
**Severity:** P2 minor
**Issue:** At the bottom of `ingest_data`, when any ingestor fails, the code raises
`SubprocessFailed(failures, returncode=1, ...)`. The `cmd` attribute is documented as a
subprocess command list, but here it receives the string list of failed ingestor names
(e.g. `["ingest-fred"]`). Callers and the Modal dashboard display `' '.join(cmd)` as the
command that failed, which is misleading — it will show `ingest-fred ingest-fdic` as if that
were a shell command.

**Fix:** Use a plain `RuntimeError` for the aggregate failure case, or introduce a separate
`IngestorsFailed` exception type:
```python
if failures:
    raise RuntimeError(
        f"ingest_data: {len(failures)} ingestor(s) failed: {', '.join(failures)}\n"
        f"Results: {summary}"
    )
```

### IN-02: `test_tail_lines_bounds_captured_output` assertion boundary is fragile

**File:** `fee_crawler/tests/test_modal_subprocess_helper.py:76`
**Severity:** P2 minor
**Issue:** The test asserts `"line 90" not in exc_info.value.stdout_tail` when `tail_lines=5`.
With 100 lines (0–99), the last 5 are lines 95–99. The assertion checks that `"line 90"` is
absent, which is correct. However, it does not assert that `"line 94"` is also absent, leaving
a gap in coverage for lines 91–94. More importantly, the string `"line 90"` is a substring
match: `"line 90"` also matches `"line 90X"` if the output happened to include such lines.
The test is not wrong but is slightly under-specified.

**Fix (optional hardening):**
```python
for absent_line in ["line 90", "line 91", "line 92", "line 93", "line 94"]:
    assert absent_line not in exc_info.value.stdout_tail
```

### IN-03: `preflight_postgres` is a GET endpoint that calls `require_postgres` — transient env issue not handled

**File:** `fee_crawler/modal_preflight.py:214-257`
**Severity:** nit
**Issue:** `preflight_postgres` calls `require_postgres()` at the top, which reads
`_DATABASE_URL` captured at **module import time** (`os.environ.get("DATABASE_URL")` at
line 24 of `db.py`). If Modal injects secrets after module load (which can happen with
certain Modal versions), `_DATABASE_URL` will be `None` even though `os.environ["DATABASE_URL"]`
would succeed. This is a pre-existing pattern throughout the codebase, but `preflight_postgres`
is the one function explicitly meant to validate the Postgres connection, making the mismatch
more confusing.

**Fix:** In `preflight_postgres`, prefer `os.environ.get("DATABASE_URL")` directly rather than
relying on the module-level `_DATABASE_URL` via `require_postgres`:
```python
db_url = os.environ.get("DATABASE_URL")
if not db_url:
    raise RuntimeError("DATABASE_URL not set; cannot run preflight_postgres")
conn = psycopg2.connect(db_url)
```
(Or confirm via Modal docs that secrets are always available at import time in their
execution model — if confirmed, this is a nit only.)

### IN-04: `daily_report.py` and `data_integrity.py` open connections without context managers

**File:** `fee_crawler/workers/daily_report.py:23`, `fee_crawler/workers/data_integrity.py:22`
**Severity:** nit
**Issue:** Both workers open `psycopg2.connect(...)` directly and never close the connection
(no `conn.close()` and no `with` statement). If an exception occurs during query execution,
the connection is leaked. In `data_integrity.py`, exceptions inside `check()` call
`conn.rollback()` but do not close the connection.

**Fix:**
```python
with psycopg2.connect(os.environ["DATABASE_URL"]) as conn:
    cur = conn.cursor(cursor_factory=psycopg2.extras.RealDictCursor)
    ...
```

Note: `psycopg2` connection objects used as context managers commit/rollback on exit but do
NOT close the connection — an explicit `conn.close()` or `with closing(conn)` is still needed
if absolute cleanup is required in short-lived worker calls.
