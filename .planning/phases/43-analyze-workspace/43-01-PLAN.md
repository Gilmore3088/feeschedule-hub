## Objective

Replace the `/pro/analyze` stub with a fully functional analysis workspace. Hamilton streams structured analysis in response to user prompts, organized into four named sections. Users can switch between analysis focus tabs (Pricing, Risk, Peer Position, Trend) to change the analytical lens. After analysis completes, 3+ follow-up prompts appear as clickable pills. Users can save analyses to workspace memory. No recommendation language appears anywhere on this screen.

---

## Success Criteria

1. All four sections (Hamilton's View, What This Means, Why It Matters, Evidence) render for any analysis focus
2. Tab switching changes analysis lens without page reload
3. Explore Further shows >= 3 context-relevant follow-up prompts
4. A saved analysis appears in the left rail on next session
5. No recommendation language or "Recommended Position" element exists in the rendered UI

---

## Threat Model

<threat_model>
**Screen:** `/pro/analyze` — authenticated Pro subscriber only

**Threats:**
- **T1 (Medium):** Prompt injection via user-supplied analysis text → Hamilton generates recommendation language despite system prompt instruction. Mitigation: client-side scan for "recommended position", "recommend you", "should set your fee" — display warning badge if detected.
- **T2 (Low):** Unauthorized access to saved analyses — user A reads user B's analyses. Mitigation: `hamilton_saved_analyses` queries always include `WHERE user_id = ${user.id}` filter. Layout already enforces premium gate via `canAccessPremium()`.
- **T3 (Low):** Cost runaway from rapid tab switching (each tab triggers a new analysis). Mitigation: daily cost circuit breaker already in `/api/research/hamilton/route.ts` ($50/day limit).
- **T4 (Low):** XSS via Hamilton-generated markdown. Mitigation: use `sanitize-html` (already in deps) or React's controlled rendering — never use `dangerouslySetInnerHTML` without sanitization.

**Residual risk:** Low. Auth gate, cost circuit breaker, and per-user DB scoping are all pre-existing.
</threat_model>

### T-43-01 — Modify Hamilton API route to support analyze mode

**File:** `src/app/api/research/hamilton/route.ts`

Add `mode` and `analysisFocus` to the request body parsing:

```typescript
const { messages, mode, analysisFocus } = await request.json();
```

When `mode === "analyze"`, append an analyze-mode system prompt suffix to the agent's system prompt before calling `streamText`:

```typescript
if (mode === "analyze") {
  const focus = analysisFocus ?? "Pricing";
  systemPrompt += buildAnalyzeModeSuffix(focus);
}
```

The analyze mode suffix (defined in `src/lib/research/agents.ts`) must:
- Instruct Hamilton to structure response into four named sections using `##` headings: `## Hamilton's View`, `## What This Means`, `## Why It Matters`, `## Evidence`
- Append `## Explore Further` with 3-4 follow-up questions as bullets
- Explicitly forbid recommendation language: "Do NOT include a recommended position, proposed fee range, or prescriptive action. The Simulate screen owns decisions."
- Anchor to the analysis focus: `You are analyzing from the ${focus} lens. Frame all insights through that perspective.`

**Acceptance:** POST to `/api/research/hamilton` with `{ messages, mode: "analyze", analysisFocus: "Risk" }` returns a stream whose content contains `## Hamilton's View` and `## Explore Further` sections.

### T-43-09 — Build AnalyzeWorkspace client shell

**File:** `src/components/hamilton/analyze/AnalyzeWorkspace.tsx`

This is the main client component that owns all state:

```typescript
"use client"

import { useChat } from "@ai-sdk/react";
import { useState } from "react";
import { ANALYSIS_FOCUS_TABS, type AnalysisFocus } from "@/lib/hamilton/navigation";
import { saveAnalysis } from "@/app/pro/(hamilton)/analyze/actions";
// ... component imports

interface AnalyzeWorkspaceProps {
  userId: number;
  institutionId: string | null;
}
```

**State:**
- `activeTab: AnalysisFocus` — current analysis focus (default: "Pricing")
- `parsedResponse` — parsed sections from stream (hamiltonView, whatThisMeans, whyItMatters, evidence, exploreFurther)
- `isSaved: boolean` — whether current analysis has been saved

**useChat configuration:**
```typescript
const { messages, input, handleInputChange, handleSubmit, isLoading, setInput } = useChat({
  api: "/api/research/hamilton",
  body: { mode: "analyze", analysisFocus: activeTab },
  onFinish: async (message) => {
    // Parse sections from completed message
    const parsed = parseAnalyzeResponse(message.content);
    setParsedResponse(parsed);
    // Auto-save if user is authenticated
    if (userId && institutionId) {
      await saveAnalysis({ ... });
      setIsSaved(true);
    }
  },
});
```

**Tab switching behavior:**
- When `activeTab` changes and there's a current prompt in the input, clear the conversation and re-submit
- If no active analysis, just update the tab (changes the prompt placeholder)

**Section parsing (`parseAnalyzeResponse`):**
- Split completed message content by `##` section headings
- Extract `## Hamilton's View` → `hamiltonView`
- Extract `## What This Means` → `whatThisMeans`
- Extract `## Why It Matters` → `whyItMatters[]` (split bullet points)
- Extract `## Evidence` → parse metrics table or key-value pairs
- Extract `## Explore Further` → `exploreFurther[]` (split bullet points)
- Fallback: if sections not found, render raw content in Hamilton's View panel

**Layout:**
```
[AnalysisFocusTabs] ← sticky top
[AnalysisResponse sections if parsedResponse]
  [HamiltonViewPanel]
  [WhatThisMeansPanel]
  [WhyItMattersPanel]
  [EvidencePanel]
[ExploreFurtherPanel] ← after completion
[AnalyzeCTABar] ← after completion, sticky bottom
[AnalysisInputBar] ← always visible, floating bottom
```

**Acceptance:**
- Tab switch updates `body.analysisFocus` in subsequent useChat calls
- After stream completes, all four section panels populate with parsed content
- Explore Further shows pills, clicking one sets `input` and submits

### T-43-10 — Wire up analyze page

**File:** `src/app/pro/(hamilton)/analyze/page.tsx`

Replace the stub with a server component that:
1. Gets current user via `getCurrentUser()`
2. Reads institution context from user record (`institution_id` or derives from profile)
3. Renders `<AnalyzeWorkspace userId={user.id} institutionId={institutionId} />`

```typescript
import type { Metadata } from "next";
import { getCurrentUser } from "@/lib/auth";
import { redirect } from "next/navigation";
import { AnalyzeWorkspace } from "@/components/hamilton/analyze/AnalyzeWorkspace";

export const metadata: Metadata = { title: "Analyze" };

export default async function AnalyzePage() {
  const user = await getCurrentUser();
  if (!user) redirect("/");

  // institution_id: use user's linked institution or empty string as default
  const institutionId = user.institution_id ?? "";

  return (
    <AnalyzeWorkspace
      userId={user.id}
      institutionId={institutionId}
    />
  );
}
```

**Acceptance:** Navigating to `/pro/analyze` renders the workspace (not the stub). Server component passes user context to client shell.
